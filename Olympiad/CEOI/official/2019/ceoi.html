1192A - Building Skyscrapers

Let's look at the problem in reverse: n times, we want to find an arbitrary non-8-articulation 4-reachable from infinity and remove it. This doesn't work if we can reach a situation where no such cell exists, so we need to prove that each 8-connected figure contains at least one non-articulation 4-reachable from infinity.

Proof: If there is a cell with 1 neighbour that is 4-reachable, we are done. Therefore, let's assume all 4-reachable (we always take this to mean "from infinity") cells have at least 2 neighbours.

For any pair of 4-adjacent cells (1,2)(c1,c2) such that 1c1 is full and 2c2 empty, the region of (1,2)(c1,c2) is the set of empty cells which are 8-adjacent to 1c1 and 4-reachable from 2c2 by only moving between these cells. The regions for fixed 1c1 can be found simply by looking at all 8-neighbours of 1c1 in cyclic order and removing full cells from this cycle; connected segments are regions.

We can now find an 8-cycle of full 4-reachable cells and a corresponding 4-path of empty cells. First, let's pick a pair (,)=(,)(cf,ce)=(cfi,cei) = (a 4-reachable full cell, an empty cell 4-reachable from infinity and 4-adjacent to it). Then, let's build a sequence of such pairs by repeating the following:

  1. Find the region R of (,)(cf,ce).
  2. Find 2 full cells that border this region and are 8-adjacent to cf (i.e. full cells that were removed to create this region); two such distinct cells must exist since cf has at least 2 full neighbours.
  3. Choose one of these cells (let's denote it by cfn) that was not chosen in the sequence immediately before cf and the 4-neighbour of cfn (let's denote it by cen) from region R. Note that this relation is bidirectional, i.e. if we start from (,)(cfn,cen), then ce is in the region of (,)(cfn,cen) and cf borders this region.
  4. Change (,)(cf,ce) to (,)(cfn,cen).

This is repeated until we reach =cfn=cfi (initial cf). The process must terminate since the number of full cells is finite. At the end, the full cells from this sequence form a simple 8-connected cycle C and the empty cells form a 4-connected path P. Each of the full cells from this cycle is therefore 4-reachable from infinity.

Now, let's compute the minimum r, minimum c, maximum r and maximum c coordinates of empty cells from cycle C. Since the cycle contains at least 2 cells, there must be a cell 0c0 which is distinct from cfi and at least one of its coordinates (,)(r,c) is one of these four extremum coordinates. We need to show that this cell is not an articulation.

It is sufficient to show that 0c0 has 3 empty 8-neighbours on one side, i.e. the 3x3 square around it looks as follows (center: 0c0, top center: empty cell 4-reachable from infinity).

If it has this form, the top 3 cells are in the same region and if we find the full cells bordering this region (among cells marked *), then these cells are on cycle C as well, since 0c0 is not cfi. Each cell marked * is 8-adjacent to one of these full cells, so emptying the central cell does not break connectivity.

W.l.o.g. let the top 3 cells in the figure above be outside the (min r, max r)(min c, max c)(min r, max r)(min c, max c) rectangle. Let's prove it by contradiction assume that there is a full cell among those three; this cell cannot be part of cycle C. Since path P consists of cells adjacent to those from cycle C, it cannot contain any cells above this figure, so the top middle cell cannot be part of path P either (it is not 4-adjacent to cfi and it is either full or has at most 1 empty neighbour in the top row). Path P and cell cfi form a 4-cycle, which splits the grid into two 8-components, one of which contains all cells from C other than cfi. Since everything above our figure does not contain any cells from +P+cfi, it must be part of the latter component, and since it is 4-connected to the top middle cell, which cannot be part of +P+cfi, the top middle cell must also be part of this component. But then, the middle cell is 4-adjacent to the second component and it is part of the first component, which is a contradiction. QED.

Hopefully, you've had enough math by now. (If not, you can try to prove the same claim for 4-connected full cells and 8-adjacency to empty cells 8-reachable from infinity.) We can finally get back to the programming problem!

Subtasks 4, 6 and 7

Checking reachability from infinity is easy. We can store the 4-connected components of empty cells, remember which component corresponds to the outside and whenever we make a new empty cell, we can update this information with the simple "merge smaller to larger" strategy. We just have to choose which empty cells to consider, since the naive estimate is (2)O(n2) empty cells; however, empty cells which have no full 8-neighbours in the original figure are not interesting, so we are left with ()O(n) interesting cells and the total time complexity of calculating connected components is (log)O(nlogn).

When is a full cell c an articulation? Let's look at all its regions that aren't isolated empty corners (since their neighbours on the border of c are 8-connected, so the following argument doesn't apply). For each pair of different regions 1r1 and 2r2, if there is no 4-path of empty cells connecting them, then for each pair of full non-adjacent 8-neighbours of c (1c1 and 2c2) such that 1,1,2,2c1,r1,c2,r2 are in this order around c, there is an 8-path of full cells connecting 1c1 and 2c2 that does not contain c. Vice versa, if there is a path connecting two such cells 1c1 and 2c2 that does not contain c, then there is no 4-path connecting 1r1 and 2r2. Since for each pair of non-corner regions 1r1 and 2r2, we can find 2 cells 1c1 and 2c2, we get an equivalent definition of a non-articulation: all non-corner regions belong to different 4-connected components of empty cells.

This simplifies things. We're already updating connected components, so the information about articulation-ness of cells is local if we know the empty 4-neighbours and connected components of a full cell, we can compute whether it is an articulation in (1)O(1) time. This information only needs recomputing if empty neighbours appear or are moved to a different component.

Now, we only need to realise that we already have to look at all empty cells that appear or are moved to a different component. It's done in the "merge smaller to larger" union-find for empty cells when merging component Ks into component Kl, we have to look at all cells from Ks to update which component they belong to, and we only create an empty cell ()O(n) times along with that. For each cell at which we look in the union-find, we can just check all its full 8-neighbours, and for each of these neighbours, we can update the information abouts its neighbours, decide whether it is an articulation and if it is reachable from the outer component, all in (1)O(1) time.

We are left with updates saying "this cell can be removed" / "this cell cannot be removed". With them, it's easy to keep the set of removable cells in amortised (log)O(logn) and choose the cell to remove in each step in (log)O(logn). The subtask type is irrelevant, we could just as well support queries "remove this cell or report that it's not allowed".

Obviously, the answer is "NO" if and only if the final configuration isn't 8-connected.

Total time complexity: (log)O(nlogn), memory complexity ()O(n).

In subtask 6, we can afford to store the graph as a 2D array, with complexity (log+)O(nlogn+rmaxcmax). Also, we are checking if a cell is an articulation in ()O(d), where =8d=8 is the number of neighbours of a cell, but less efficient implementations like finding regions and checking all pairs of regions in (2)O(d2) become possible.

In subtask 4, we only need to determine if it's possible to remove a skyscraper in (1)O(1). Whenever we're removing a skyscraper, we can simply find the last removable one, then update the connected components, either by merging existing ones or running BFS on the interesting cells. This works in (2)O(n2).

Subtasks 3 and 5

When we are allowed to choose the output sequence, a much simpler approach becomes possible:

  • First, build an arbitrary skyscraper.
  • Store the set of skyscrapers which haven't been built, but are 8-adjacent to those which have been built.
  • While we need to build a skyscraper, pick the rightmost one from this set (with the maximum ci); if there is more than one such skyscraper, pick the one with the maximum ri. Then, update the set.

This takes (log)O(nlogn) time when the skyscrapers are stored efficiently (e.g. in an STL set<>), or (2)O(n2) by brute force.

The set of built skyscrapers is 8-connected, but it isn't immediately clear why we can't fail by closing an 8-cycle and then building a skyscraper inside it. However, it makes sense informally, before building the last skyscraper from this cycle, we can already build everything on its inner border, and if the final configuration is 8-connected and there is something inside the cycle we still need to build, there will be a skyscraper we haven't built yet on this border with greater ci (or equal ci and greater ri). We'll build everything inside a cycle before we finish the cycle itself.

Subtask 2

We can use the approach for =1t=1, or we can guess that the hypothesis about existence of a removable skyscraper is true. Then, we can choose a skyscraper to remove by checking which ones can be removed (by removing and running BFS on the remaining graphs of empty and full cells). If we do this n times, the time complexity is (3)O(n3).

Subtask 1

We can try all (!)O(n!) possible orders and check if they are valid.

1192B - Dynamic Diameter

Subtask 1

In the first subtask the limits are small enough to do literally what is written in the problem statement after each update we run DFS from each vertex to compute all pairs of distances and report the largest one.

Complexity (2)O(qn2).

Subtask 2

In the second subtask, we need to compute the diameter in ()O(n). There are two ways:

  • The first one is a somewhat standard trick. Use DFS from vertex 11 to find the most distant vertex, let's call it u. The diameter is then the distance to the most distant vertex from u, which can be again found using DFS.
  • The second approach is using a dynamic programming on a tree, which will be easier to generalise. Root the tree in vertex 11. For each vertex v, compute Dv the maximum distance to leaf in a subtree of v. This can be done in ()O(n) recursively. Then the answer is just maximum (through all vertices u) of Du and max12 are children of (1+2+,1+,2)maxv1v2 are children of u(Dv1+Dv2+wu,v1+wu,v2)

Complexity ()O(qn).

Subtask 3

When the graph is a star, it means that the diameter consists only of two edges, both of which are incident to vertex 11. We can thus simply maintain the weight of each edge in a multiset and output the sum of the two largest values as an answer. Be careful about =2n=2.

Complexity ((+)log)O((q+n)logn)

Subtask 4

In the fourth subtask, we modify the DP from the second subtask slightly by introducing a new value Su to be the diameter of the subtree rooted at u. We can see that

=max(,max child of ,max12 children of 1+2+,1+,2)Su=max(Du,maxv child of uSv,maxv1v2 children of uDv1+Dv2+wu,v1+wu,v2)

and the diameter equals 1S1.

Upon changing the weight of the edge {,}{u,v}, only the values ,Sx,Dx where x is an ancestor of u need to be updated. As each update can be processed in constant time (the number of immediate children of each vertex is at most 22) and as the tree is balanced, it's depth is (log)O(logn).

Overall this needs (+log)O(n+qlogn) time.

Subtask 5

Assume that the diameter always goes through vertex u. This means that the diameter is found by picking two children of vertex u 1v1 and 2v2 such that 1+2+,1+,2Dv1+Dv2+wu,v1+wu,v2 is maximised.

To calculate each value of Dvi, we can use a segment tree that maintains the distances of each vertex to root. When the vertices are ordered using DFS, an update becomes an addition on a range, so any segment tree with lazy propagation is sufficient.

To calculate the maximum of the sum above, we use the approach outlined in subtask 3 we maintain a multiset of +,Dvi+wu,vi and pick the largest two values.

In subtask 5, we can simply assign =1u=1. This needs (log)O(qlogn) time.

Subtask 6

In subtask 6 we cannot pick any single vertex through which all the diameters traverse. Nevertheless, we can still pick a vertex u and find the longest path that goes through vertex u to get a lower bound on the answer. Then, we remove u from the tree, leaving us with some number of smaller trees. This approach can be recursively applied to them.

Performed naively, this leads to a quadratic performance for example consider a path on n vertices, where we always select a leaf as the root. We need to limit the depth of this recursion in some way.

We do this by always picking the centroid as a root. Recall that centroid is a vertex that when removed splits a tree into a number of subtrees each having at most /2n/2 vertices where n is the size of the original tree. Every tree has 11 or 22 centroids that can be found in linear time using a simple DP.

If we always pick a centroid, then in (log)O(logn) iterations we end up with a couple of trees of size 11, for which the answer is 00, so this can be done in (log)O(nlogn).

Now we simply maintain the segment tree for each centroid. As each edge is in (log)O(logn) trees, each update can be performed in (log2)O(log2n). The overall complexity is ((+)log2)O((n+q)log2n).

Subtask 6 alternative approach

Let's root the tree. Recall the trick from subtask 2: To compute the diameter, we should find the deepest (most distant from root) vertex of the tree let's call it u and then find the maximum distance from u. Since we cannot expect anything about u, this problem can also be formulated as just finding the farthest vertex from a given vertex.

Instead of centroid decomposition, we can use heavy-light decomposition. For each vertex v, there is at most one son s connected to v by a heavy edge. Let mv denote the maximum of depths of all descendants of v which are not descendants of s, inclusive. On each path of the HLD, we should build a segment tree which stores the values 2mv2dv, where dv is the depth of v. Also, let's measure these depths from the top of the path containing v instead of from the root.

When looking for the maximum distance from vertex u, let's move to the root and look at all ancestors of u (including u itself). When we take the son s from which we reached v and compute the maximum m of depths of descendants of v which aren't descendants of s (if =v=u, there are no such vertices because s doesn't exist), then the maximum of lengths of all paths from u which have v as the LCA is +2m+du2dv, where the depths are again measured from the top of the path containing v. The maximum distance is the maximum of all these lengths and it's easy to extend the process that computes it to find the vertex with this distance.

Of course, we can't afford to do that by brute force, but HLD helps. If the last edge we traversed in order to reach an ancestor v was a heavy edge, then =m=mv. This isn't the case only when we just reached a new heavy path; there are (log)O(logn) heavy paths on any path to the root and therefore just (log)O(logn) such vertices. When we're moving from u to the root and reach a path p in a vertex v, we need to compute +2m+du2dv for this vertex. Then, we know that we must visit all vertices v above v on this path, so the maximum of +2mv+du2dv for all these vertices is a prefix maximum from the segment tree of path p, plus du.

How to compute m? With preorder numbering of the vertices, each subtree becomes a segment, so we just need a segment tree that stores depths from the root and m is the maximum from two segments minus the depth of the top of the current path. This segment tree lets us find depths of individual vertices as well.

Now, we just need to handle updates. When we add a to the weight of an edge from u to v (v is a son of u), we add a to the depths from the root of all vertices in the subtree of v. The values of m can only change for u and the ancestors of u which are reached by a light edge. There are only (log)O(logn) of them, so we can afford to recompute m for each of them. Recall that we're storing 2m2d in the tree, so we need to also subtract 22a from all vertices on the path containing the updated edge, located below that edge (d changes only for these vertices this is why we aren't using depths from the root). Therefore, the segment trees built over HLD paths need to support operations "update value", "add to segment" and "get maximum from segment".

The time complexity of this approach is (+log2)O(n+qlog2n), since we're handling (log)O(logn) paths per query and doing (1)O(1) segment tree operations per path.

1192C - Cubeword

In all subtasks we solve the problem independently for each possible length of a word and sum the answers.

To handle palindromes and words that are reverse of each other, we first add the reverse of each word to the dictionary and then remove duplicates. Then, we will fix a direction for each edge and we will ask that the word written along the edge in that direction has to be in the new wordlist. (This is clearly equivalent to the original problem, and it allows us to choose the directions of edges in any way that suits our approach.)

For each pair of letters a, b precompute the number of words that begin with a and end in b.

Subtask 1

We brute-force all combinations of letters in the vertices of the cube and use the precomputed values to find how many cubewords are there with these letters fixed. Complexity (+||8)O(n+||8).

Subtask 2

Label the vertices 11 through 88 with vertices 11 through 44 lying around the bottom face and vertex +4i+4 directly above vertex i. We will assign letters to vertices in the increasing order of their ids. Start by assigning letters to vertices 11 through 44 and count the number of possibilities of assigning letters to the edges on the bottom face. We then try all possibilities of assigning a letter to vertex 55, and account for the edge {1,5}{1,5}. Note that all edges incident to the vertex 11 are already processed, hence we may now drop the value of 11 from our state. More generally, after processing vertex j we can forget the letter in vertex 4j4.

This forms a dynamic programming solution with 8||48||4 states, where each state has |||| transitions, each of which can be performed in constant time.

Complexity (+||5)O(n+||5).

Subtask 3

When viewed as a graph, the cube is bipartite with both partitions of size 44. Consider all possibilities of assigning the letters to one partition. We want to process each of the remaining vertices in (1)O(1).

Precompute [][][]D[a][b][c] as the number of triples of words such that they all begin with the same letter and the other end with a,b,c, respectively. This precomputation can be done naively in (||4)O(||4).

Coming back to our problem, let the letters on one partition be a,b,c,d respectively. Observe that the answer is [][][][][][][][][][][][]D[a][b][c]D[a][b][d]D[a][c][d]D[b][c][d].

Total complexity is (+||4)O(n+||4).

Fun fact: Geometrically, this solution can be viewed as follows. A cube can be cut into five tetrahedrons: one that's completely inside and four such that each of them contains one corner of the cube (and thus three edges). In this solution you brute force the letters in the corners of the inner tetrahedron, and once you fix those, each of the remaining tetrahedrons is independent of the other three.

Subtask 4

To reduce the complexity even further, note that a cube can be rotated and flipped in 4848 different ways: there are 66 possibilities for the bottom face, after the bottom face is fixed, there are 44 options for the front face, and after that there are two possibilities on the left face. In exactly half of the rotations, the vertex partitions swap places (for instance when rotating the cube by 90 degrees in any principal axis) and in the second half they stay in position, possibly permuting the vertices (for example rotation the cube by 180 degrees, or rotating the cube by 90 degrees and then mirroring it).

Assume for a moment that the letters a, b, c, d on the vertices of the first partition are guaranteed to be distinct. Then there is a way of rotating and flipping the cube in such a way that <<<a<b<c<d. Hence we can calculate the answer only for this case and multiply it by 2424.

However, the letters a, b, c, d are not necessarily distinct. To mitigate this, we simply consider all possibilities of the form abcd, and multiply the answer of each of them by the number of permutations of (,,,)(a,b,c,d). This reduces this step of the solution to (||4/24)O(||4/24).

Precomputing the array D can also be optimised we again consider only cases abc and use the fact that [][][]=[][][]==[][][]D[a][b][c]=D[a][c][b]==D[c][b][a]. The complexity of this step is thus reduced to (||4/6)O(||4/6).

To summarise, we are able to reduce the complexity 66-fold by considering symmetries.

Imho if someone can figure out a full solution just with the help of subtask structure in-contest, that's fine and that contestant deserves the points just as much as those who wrote HLD from memory.

      I don't see A as particularly implementation-heavy. If you have the main idea (that you need to write one function to locally decide removability and how to do that), which is the hard part, everything else is quite simple. My code is written for clarity and some efficiency rather than code golf and it has "just" 300 lines.

      1. You need to parse the input into a graph with 99N vertices and do a BFS/DFS to find components. There's no way around this, but it's all very easy.
      2. You need to write the "update" function. All decisions about articulations and reachability are contained here. It uses graph information: components, degrees, which cells are empty and the graph itself.
      3. When you remove a cell, you need to merge components and update the degrees. This is classic union-find. In the process, with the slower version of union-find where you don't maintain a tree of merges, but just merge smaller components to larger ones, you're looking at all vertices whose component changed. Look at their neighbours and run update() for them. Also run update() for neighbours of the cell you just emptied.
